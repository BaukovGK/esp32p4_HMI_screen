# ТЗ: переделка UI HMI-панели RO-установки

> **Статус:** утверждено к реализации (на базе ответов в `UI_DESIGN_BRIEF.md`).
> **Целевая платформа:** Waveshare ESP32-P4-NANO + 10.1″ IPS 1280×800 (landscape).
> **Стек:** ESP-IDF 5.4.4 + LVGL 9.2 + esp_lvgl_port.
> **Связь с контроллером:** MQTT через хаб (Mosquitto на Orange Pi).

## 1. Общие положения

### 1.1 Целевая аудитория и роли

| Роль | Доступ |
|---|---|
| **Гость** (без авторизации) | Только просмотр главного экрана |
| **Технолог** (по PIN-коду) | Полный доступ ко всем экранам, настройкам, командам |

PIN-код **числовой** (4-6 цифр), хранится в NVS, задаётся в настройках при первой авторизации.
Сессия активна **N минут после последней активности** (по умолчанию 10 мин, настраивается).

⚠️ **Критичное:** блокировка экрана **НЕ** останавливает текущие процессы установки.
Контроллер продолжает работу в AUTO/WASHING без UI.

### 1.2 Принципы дизайна

1. **Лаконичность** — один экран = одна задача. Нет «дашбордов на всё».
2. **Высокая читаемость** — большой шрифт (минимум 16px), tabular-numerics для чисел, цветовая индикация состояний.
3. **Touch-friendly** — все интерактивные элементы минимум 60×60 px.
4. **Подтверждения** — все опасные команды через modal «Подтвердите действие».
5. **Цветовая семантика** — зелёный=ok, жёлтый=warning, красный=critical/alarm, синий=accent/info.
6. **Стиль** — промышленный минимализм. Не «3D-кнопочная SCADA», а близко к Apple HIG / Material 3.

### 1.3 Темы и языки

| Параметр | Default | Опции |
|---|---|---|
| Тема | **Light** | Light / Dark (referenced в `proto/темная тема.png`) |
| Язык | **RU** | RU / EN |
| Яркость подсветки | **80%** | 0..100% (NVS persist, регулируется в настройках) |

Переключение темы и языка — оператор может в любой момент через status bar.

### 1.4 Брендинг

Цветовая палитра ориентируется на корпоративный стиль из `proto/цвета.png` (НТТ — «Новые Трубные Технологии»):
- **Primary accent:** тёмно-зелёный (промышленный, не яркий)
- **Backgrounds:** светлый/тёмно-серый
- **Splash screen** — НЕ нужен (per Q9 «не делаем»)

Подробная палитра — `UI_TOKENS.md`.

Это 4 экрана покрывают топ-3 сценария Q1.3:
- **«Авто запуск»** — с главной (1 тап + confirm)
- **«Ручной запуск»** — с главной (1 тап + confirm) → перевод в MANUAL
- **«Сохранение установок»** — через Настройки (4 категории + Save)

Плюс жизненно важный 4-й сценарий «реакция на аларм» — экран Алармы.

### 2.2 Поведение алармов поверх любого экрана

При появлении нового аларма:
1. **Toast-banner** в status bar изменяется (текст аларма + цвет по категории).
2. **Pulse-индикатор** на вкладке «Алармы» (счётчик активных).
3. **Модальное окно** при критических алармах (CRITICAL/ALARM): «Аварийная ситуация: ...», 2 кнопки [Перейти к алармам] [Закрыть]. Закрытие НЕ заглушает аларм, только убирает модал.
4. **Звуковой сигнал** через ES8311 кодек (короткий beep на каждом новом, повторяется раз в 30с пока активный).

См. подробно в `UI_ALARM_MATRIX.md`.

## 3. Layout базовых элементов

### 3.1 Status bar (top, 44 px)

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│ HH:MM:SS │  Сообщение системы (или активный аларм)              │ ●Хаб ☾ RU PIN│
└─────────────────────────────────────────────────────────────────────────────────┘
```

| Элемент | Что | Поведение |
|---|---|---|
| Время | HH:MM:SS, обновление 1/с | синхронизация с RTC контроллера через MQTT |
| Сообщение | текущее состояние или активный аларм | цвет меняется (info/warn/danger) |
| ●Хаб | индикатор связи с MQTT broker | зелёный=ok, красный=offline |
| ☾ / ☀ | переключатель темы | мгновенно меняет CSS variables |
| RU/EN | переключатель языка | мгновенно re-translate |
| PIN | состояние авторизации: 🔓=открыта/🔒=закрыта/🚫=нет PIN | tap → диалог входа/выхода |

### 3.2 Tab bar (bottom, 64 px)

7 вкладок (по 1280/7 = ~183 px каждая, touch-friendly):

```
┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│   ⌂      │   ⟲     │   ☰      │   ⚠      │   ⚡     │   ⚒      │   ⚙      │
│ Главная  │ Промывка │Параметры │ Алармы   │ Энергия  │Диагност. │Настройки │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

Иконки — простые символы Unicode или LVGL built-in symbols. **Pulse-badge на «Алармы»** при активных событиях (счётчик в красном кружке).

### 3.3 Quick actions (всегда видны на любом экране)

В status bar рядом с PIN-индикатором:
- **🚨 АВАРИЙНЫЙ СТОП** — большая красная кнопка справа от status bar.
  **Один тап → команда `ro_plant/command/mode = stop` сразу** (без confirm).
  Стоп — защитное действие, ложный тап не вреден (можно сразу запустить заново).
- **🔇 Заглушить** — рядом с E-STOP, видна только если есть активные алармы.

### 3.4 Content area (между bars, 692 px высоты)

Свободное пространство для контента экрана. Все экраны должны помещаться без скролла, **кроме**:
- Параметры (таблица всех датчиков может прокручиваться)
- Алармы (история — list со скроллом)
- Журнал/отчёты

Для прокручиваемых областей — крупные scroll-индикаторы (минимум 8 px ширины), большие touch-targets для list items (минимум 56 px).

## 4. Ключевые сценарии (90% времени)

### 4.1 Сценарий «Смотрю состояние» (Q1.3 #1)

Оператор смотрит на главную из коридора. Видит:
- На мнемосхеме — текущие потоки (активные/неактивные трубы цветом).
- Статусы насосов (выключен/запускается/работает/ошибка) в виде badge.
- Уровни в ёмкостях (поплавки DI1-DI4).
- 4-6 ключевых KPI справа: P3, P4, σ-пермеат-2, расход пермеата, мощность ВД.

Всё видно за 1 взгляд. Никаких действий не нужно.

### 4.2 Сценарий «Запуск промывки» (Q1.3 #2)

Промывка — **интерактивная многошаговая** последовательность с подтверждениями оператора между фазами.

```
1. [PIN] Перейти на вкладку «Промывка»
2. [Кнопка] «Начать цикл» → confirm → команда WASHING контроллеру
3. Фаза 1: НАГРЕВ — преднагнетания + нагреватель ВКЛ, ждём 30 мин или t°≥35°C
   → На UI: phase track, текущая температура vs целевая, обратный таймер
4. По достижении t° → toast «Готово к подаче» → [Кнопка] «Подтвердить подачу»
   (предварительно оператор переключает вентили физически)
5. Фаза 2: ПОДАЧА — циркуляция реагента, ~20 мин
6. По завершении → [Кнопка] «Подтвердить слив»
7. Фаза 3: СЛИВ — оператор открывает дренаж физически, подтверждает
8. Возврат в IDLE
```

Каждая фаза имеет состояния: **активна / ожидает оператора / завершена / прервана**.
Кнопка «Отмена цикла» доступна на любом шаге (с double-confirm).

### 4.3 Сценарий «Реакция на аларм»

(Не отмеченный явно в Q1.3, но критичный):

1. **Появляется аларм** → toast в status bar + звуковой сигнал.
2. Если CRITICAL/ALARM — модал «Аварийная ситуация».
3. Оператор: [Перейти к алармам] → экран алармов.
4. На экране — карточка с описанием, временем, рекомендацией.
5. **Действия**:
   - 🔇 Заглушить (по PIN) — отправляет `ro_plant/command/silence`, аларм остаётся активным но звук/мигание прекращаются.
   - ✓ Сбросить (по PIN) — для FAULT-состояний, отправляет команду перехода в IDLE.

История событий показывается ниже активных, **до сброса всего журнала** (отдельная команда «Очистить» по PIN).
**TODO API:** добавить команду `ro_plant/command/alarm_clear` для сброса истории на стороне контроллера.

## 5. Состояния, темы, поведение

### 5.1 Глобальные состояния системы (из контроллера)

| Состояние | Цвет statusbar | Цвет акцентов мнемосхемы |
|---|---|---|
| `IDLE` | серый | приглушённый |
| `AUTO/RUNNING` | зелёный | зелёный (живой) |
| `AUTO/STARTING` | синий | синий (анимация запуска) |
| `WASHING/*` | оранжевый | оранжевый |
| `MANUAL` | синий | синий |
| `FAULT` | красный (мигающий) | красный |

### 5.2 Состояния насосов (badge на мнемосхеме)

| Состояние | Источник | Badge |
|---|---|---|
| Выключен | `RO* = 0` (DO mask) | серый ⏸ OFF |
| Запускается | `RO* = 1` И ramp-up период (DI* ещё 0) | синий ▶ START |
| Работает | `RO* = 1` И `DI* = 1` | зелёный ● RUN |
| Ошибка | таймаут DI или ALARM_PUMP_*_NO_CURRENT | красный ✕ ERR |

### 5.3 Поведение при потере связи с контроллером

`ro_plant/availability = offline` (LWT topic):

**Релизная версия:**
- Главный экран → splash «Нет связи с хабом, последние данные на HH:MM».
- Tab bar заблокирован (тапы игнорируются).
- Кнопки команд disabled.
- Через 60 секунд — попытка автоматического переподключения, индикатор «Восстанавливаю...».

**Тестовая версия (DEBUG build):**
- Лог в status bar «Связь потеряна», но UI не блокируется.
- Все кнопки активны, отправляют команды (которые могут не дойти).
- Это позволяет тестить UI без реального контроллера.

Контроль через `CONFIG_HMI_BLOCK_ON_OFFLINE` Kconfig flag.

### 5.4 Edge cases

- **Первый запуск:** сразу рабочий экран с дефолтами. Без wizard. PIN не задан — статус 🚫.
- **Stale data** (last_msg_time_us > 5 секунд назад) — числа становятся серыми, добавляется иконка ⏳ рядом с KPI.
- **Out-of-range value** — KPI красный + pulse, sensor-fault значение `null` → показывается «—».
- **API mismatch** (новый Controller, старый HMI) — не релевантно. API согласован, обновляются вместе.

## 6. Технические требования

### 6.1 Производительность

| Параметр | Цель | Жёсткий минимум |
|---|---|---|
| Частота обновления экрана | 60 Hz | 20 Hz |
| Latency от MQTT → UI | <500 мс | <2 с |
| Время отклика на тап | <100 мс | <300 мс |
| Boot-time до главного экрана | <8 с | <15 с |

### 6.2 Тренды

- Длительность: **3 часа** на каждый параметр.
- Sample rate: **1 точка / 5-10 секунд** (≈1080-2160 точек на 3ч).
- Buffer: ~5-10 KB на канал × 11 каналов (4 P + 4 Q + 3 σ) = 60-110 KB в PSRAM.
- **БЕЗ persistence в NVS** — при reboot тренды теряются.
- График — простая X/Y линия, без zoom/pan.

### 6.3 Audit log (Q7.4)

Все действия оператора публикуются в `ro_plant/hmi/audit`:
```json
{
  "ts": 1234567890,
  "action": "start_auto" | "stop" | "washing_start" | "settings_save" | "silence" | "pin_unlock" | ...,
  "user": "guest" | "tech",
  "extra": { ... опциональные параметры ... }
}
```

### 6.4 OTA (Q7.3)

OTA НЕ из UI. Обновление через:

## 9. Дизайн мнемосхемы (стандарт ISA-5.1 P&ID)

В прототипе и в финальной реализации мнемосхема рисуется по правилам
**ANSI/ISA-5.1 — Instrumentation Symbols and Identification** (стандарт для
схем КИПиА). Это даёт промышленному оператору привычный вид и сокращает
обучение.

### 9.1 Элементы

| Элемент | Стандарт | Реализация в прототипе |
|---|---|---|
| **Ёмкости** | Прямоугольник со скруглением + заливка по уровню воды | `<rect class="tank">` + `<rect class="tank-water">` |
| **Насосы** | Прямоугольник + цветовое кодирование (RUN/OFF/ERR) | `<rect class="equipment-active">` (зелёный) или `equipment-bg` (серый) |
| **Мембраны** | Прямоугольник с разделительной линией внутри | `<rect>` + `<line dasharray>` (показывает мембранный слой) |
| **Приборы (датчики)** | **Круг** диаметром ≈11–13 мм с тегом сверху и значением снизу | `<g class="sensor-circle">` с `<circle r="22">` (44 px) |
| **Точки врезки** | Маленький filled circle на трубе | `<circle class="sc-tap" r="3.5">` |
| **Leader-line** | Тонкая линия от прибора до tap-point | `<line class="sc-leader" stroke-width="1">` |
| **Junctions** | Filled circle серого цвета на разветвлении | `<circle class="pipe-junction" r="4">` |
| **Активный поток** | Линия с анимацией движения воды | `<line class="pipe-active">` + `<line class="pipe-flow">` overlay |
| **Неактивный** | Серая сплошная линия | `<line class="pipe">` |
| **Рецикл/концентрат** | Бирюзовая dashed-линия | `<line class="pipe-recycle">` (с анимацией если flowing) |

### 9.2 Анимация движения потока

Используется вместо стрелок направления:
- **Активные трубы** — поверх сплошной линии накладывается полупрозрачный
  dashed-overlay (`pipe-flow`). Анимация `stroke-dashoffset: -16` за 1.4 с
  создаёт иллюзию «бегущих капель воды» в направлении потока.
- **Рециклы** — анимируется напрямую stroke-dashoffset существующего dashed
  (`pipe-recycle.flowing`), 1.6 с за цикл.
- **Реверс** — для row 2 (поток справа налево) применяется
  `.reverse` модификатор с обратной анимацией.

⚠️ Анимация **отключается** автоматически при `prefers-reduced-motion: reduce`
(стандарт WCAG 2.1 AA). На LVGL — глобальный флаг `s_reduce_motion`.

### 9.3 Локализованные рециклы

Рециклы 1-й и 2-й ступени реализованы как **компактные петли вокруг
соответствующих насосов** (а не длинные обходы по всей мнемосхеме):

```
Рецикл 1ст: над row 1 (петля Y=100..210, через зону между мембраной 1 и насосом 1)
Рецикл 2ст: под row 2 (петля Y=350..470, через зону между мембраной 2 и насосом 2)
```

Подключение к main pipes — через **junctions на трубопроводе**, не на крае
блоков. Между блоком и junction есть короткая `pipe-active` outlet-труба
(60 px для рецикла 1ст, 30 px для рецикла 2ст) — показывает что
концентрат покидает мембрану по своей трубе.

### 9.4 Приборные индикаторы — состояния

| Состояние | Рамка круга | Leader/tap | Значение | Триггер |
|---|---|---|---|---|
| **OK** | accent (зелёный/тёмно-зелёный) | accent | text-primary | value < warn_at |
| **Warn** | warning (жёлтый) | warning | warning | warn_at ≤ value < alarm_at |
| **Danger** | danger (красный) + pulse-anim | danger | danger | value ≥ alarm_at |
| **Offline** | text-muted (серый dashed) | dashed | «—» | sensor fault или нет данных |

Все 4 элемента группы (`.sensor-group .sc-frame .sc-divider .sc-leader .sc-tap`)
перекрашиваются согласованно при смене состояния.

## 10. Интерактивность мнемосхемы

### 10.1 Клик на прибор → детальный модал

При клике на любой приборный круг (sensor-circle) на мнемосхеме
открывается **модальное окно с подробной информацией о датчике**.

**Содержимое модала:**

| Секция | Что показывает |
|---|---|
| **Заголовок** | Тэг + полное название (RU/EN) + место установки в установке |
| **Текущее значение** | Крупно (64 px) с единицами + бейдж состояния (OK/Warn/Danger/Offline) |
| **Range bar** | Шкала всего диапазона датчика с цветовыми зонами (зелёная 0-70%, жёлтая 70-90%, красная 90-100%) + указатель текущей позиции |
| **Тренд за 3 часа** | Мини-график (~60 точек, 1 точка / 3 минуты) с area-fill |
| **Статистика тренда** | Мин / Среднее / Макс за период |
| **Уставки** | Таблица: «Норма», «Предупреждение», «АВАРИЯ» с пороговыми значениями (warn_at, alarm_at из конфига) |
| **Источник** | Modbus slave + регистр/канал; время последнего обновления |

**Взаимодействие:**
- Открытие — клик на круг прибора (cursor:pointer + hover-подсветка)
- Закрытие — кнопка [×] в углу, кнопка [Закрыть] внизу, клик по
  затемнённому фону (backdrop click)

**Языки:** при смене RU/EN названия и место установки переводятся через
словарь `sensorMeta[tag].name_{ru,en}` и `location/loc_en`.

### 10.2 Sensor metadata

Конфигурация приборов хранится в коде (не в NVS, не настраивается
оператором). На прототипе — словарь `sensorMeta` в `proto/app.js`,
в LVGL — `static const ui_sensor_meta_t s_sensors[]` массив.

**Поля метаданных:**

```c
typedef struct {
    const char *tag;             // "P3"
    const char *name_ru;         // "Давление после насоса 1-й ступени"
    const char *name_en;         // "Pressure after stage-1 pump"
    const char *location_ru;     // "между насосом 1-й ст. и мембраной 1-й ст."
    const char *location_en;
    const char *unit;            // "bar"
    float range_min, range_max;  // 0, 40
    float warn_at;               // 32 (NaN если порога нет)
    float alarm_at;              // 35 (NaN если порога нет)
    const char *modbus_source;   // "Slave 1 · AI3 · 4–20 mA"
    plant_data_field_t source;   // на какой ke в plant_data ссылаться
} ui_sensor_meta_t;
```

**Текущие приборы (10 штук на мнемосхеме):**

| Тэг | Range | Warn | Alarm | Где |
|---|---|---|---|---|
| P1 | 0–6 bar | 5.0 | 5.5 | После преднагнетания |
| P2 | 0–6 bar | 4.5 | 5.0 | После фильтра |
| P3 | 0–40 bar | 32 | 35 | После насоса 1-й ст. |
| P4 | 0–10 bar | 7.5 | 8.0 | После насоса 2-й ст. |
| Q1 | 0–5 м³/ч | — | — | Питательная |
| Q2 | 0–3 м³/ч | — | — | Рецикл 1ст |
| Q3 | 0–3 м³/ч | < 0.8 | — | Пермеат товарный |
| Q4 | 0–2 м³/ч | — | — | Рецикл 2ст |
| σ2 | 0–200 мкСм | 30 | 50 | Пермеат 1ст |
| σ3 | 0–50 мкСм | 5 | 10 | Пермеат товарный (товарное качество) |

⚠️ σ4 (концентрат) **не на мнемосхеме**, доступен через экран
«Параметры» — подключение опционально через настройки.

### 10.3 Тренды (3 часа × 10 датчиков)

**Размер буфера:** 60 точек × 4 байта (float) × 10 датчиков = **2.4 КБ**
в PSRAM на полные тренды для всех приборов мнемосхемы.

**Sample rate:** 1 точка / 3 минуты (180 секунд). При 60 точках на
3-часовое окно — старые точки замещают новые в кольцевом буфере.

**Persistence:** **НЕ сохраняется в NVS** — теряется при reboot.
Это согласно §6.2 (компромисс памяти/сложности).

**LVGL компоненты:**
- `lv_chart_create` с `LV_CHART_TYPE_LINE` или `_AREA`
- Один series на датчик
- Обновление — `lv_chart_set_next_value` каждые 3 минуты
